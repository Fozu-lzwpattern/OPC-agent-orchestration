# 任务分解方法论

## 三层结构：OKR → Epic → Task

```
Objective: {一句话目标}
├── KR1: {可衡量的关键结果}
│   ├── Epic 1.1: {大功能块}
│   │   ├── Task: {具体任务} → 角色 [依赖] [预算]
│   │   └── Task: ...
│   └── Epic 1.2: ...
└── KR2: ...
```

## Task 定义规范

```yaml
task_id: T001
name: 确定活动主题
epic: E1.1-活动策划
role: 活动策划员
depends_on: []           # 依赖的 task_id 列表
estimated_tokens: 25000  # 预估 token
status: pending          # pending/assigned/in_progress/review/completed/failed
priority: high           # high/medium/low
output: markdown         # 产出格式
```

## 状态机

```
pending → assigned → in_progress → review → completed
                         ↓            ↓
                       failed       failed
```

## 依赖管理

1. **硬依赖**：必须等上游完成。如"搭建"依赖"策划方案"
2. **软依赖**：最好有但可以先开始。如"内容"软依赖"设计风格"
3. **无依赖**：可并行执行

## 拆分粒度原则

- 每个 Task 预估 token < 50K
- 每个 Task 产出明确可验收
- 每个 Task 由单一角色完成
- 依赖链不超过 5 层

## 示例：营销活动

```
Objective: 315 促销活动上线
├── KR1: 活动页面 3.10 前发布
│   ├── Epic 1.1: 策划
│   │   ├── T001: 确定主题人群 → 策划员 [无依赖] [25K]
│   │   └── T002: 制定预算分配 → 预算员 [T001] [20K]
│   ├── Epic 1.2: 搭建
│   │   ├── T003: 创建高达活动 → 搭建员 [T001] [30K]
│   │   ├── T004: 配置页面组件 → 搭建员 [T003] [40K]
│   │   └── T005: 配置供给数据 → 供给员 [T003] [30K]
│   └── Epic 1.3: 发布
│       ├── T006: 发布诊断 → 发布员 [T004,T005] [15K]
│       └── T007: 正式发布 → 发布员 [T006] [10K]
└── KR2: ...
```

## 任务拆解四问（v5.1 强制检查）

> CEO 每拆出一个 Agent，必须能回答这四个问题。
> 回答不出来 → 任务没拆清楚，继续拆，不要 spawn。

```
Q1. 这个 Agent 的「唯一产出」是什么？
    ✅ 好：workspace/reports/2026-03-19-topic-a.md
    ❌ 差：帮我研究一下 X

Q2. 产出的验收标准是什么？
    ✅ 好：行数 > 80，包含"核心发现"章节，npm test 全绿
    ❌ 差：写得好

Q3. 这个 Agent 依赖什么输入？
    ✅ 好：workspace/reports/topic-a.md（由 researcher_a 产出）
    ✅ 好：无（独立任务，可直接 spawn）
    ❌ 差：（空白/未声明）

Q4. 文件边界是什么？（owned_paths）
    ✅ 好：packages/core/src/
    ❌ 差：（未声明，默认全局 → 并行风险）
```

## 并行判断规则（v5.1 明确化）

### 可以并行的条件（全部满足才并行）

```
✅ 两个 Agent 的输入来源互不依赖（A 不需要读 B 的产出）
✅ 两个 Agent 的 owned_paths 不重叠
✅ 两个 Agent 的产出不需要互相参考
```

### 必须串行的条件（任意一条 → 强制串行）

```
❌ Agent B 的任务描述里包含"基于上一步"/"参考前一个 Agent"
❌ 两个 Agent 可能写同一个文件（哪怕一个是主写一个是"微调"）
❌ Agent B 的输入是 Agent A 的产出
```

### 典型模式速查

| 场景 | 模式 | 说明 |
|------|------|------|
| 多研究员各研究不同方向 | 并行 ✅ | 输入独立，输出独立 |
| 研究员 + 汇总员 | 串行 ❌ | 汇总员依赖研究员产出 |
| 前端 + 后端分离开发 | 并行 ✅ | 文件边界清晰不重叠 |
| 编码 + 测试（同一模块）| 串行 ❌ | 测试依赖编码产出 |
| 多会场搭建（各自独立Tab）| 并行 ✅ | 操作对象完全独立 |

## 拆解粒度校验（v5.1）

拆出的每个 Task 必须通过以下检验：

```
☑ 预估 token < 80K（超过则继续拆分）
☑ 产出是单一文件/目录（不是"一堆东西"）
☑ 由单一角色完成（不是两个角色混着干）
☑ 依赖链层数 ≤ 5（超过则重新规划结构）
☑ 四问全部能回答（见上方四问框架）
```

**黄金原则**：一个 Agent = 一个可独立验收的产出单元。
